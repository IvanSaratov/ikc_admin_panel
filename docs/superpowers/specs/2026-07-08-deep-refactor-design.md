# Дизайн глубокой поэтапной перестройки приложения

## Контекст

Проект уже имеет рабочую структуру modular monolith: доменные пакеты лежат в
`backend/*`, SQL генерируется через `sqlc`, UI рендерится через `templ`, а
SQLite-миграции являются источником правды для схемы. Рефакторинг нужен не для
спасения хаотичного кода, а для того, чтобы приложение можно было спокойнее
развивать дальше: добавлять новые рабочие процессы, сопровождать production
инсталляции и читать код без знания истории всех этапов MVP.

Главный выбранный подход: глубокая перестройка через вертикальные этапы. После
каждого этапа приложение должно компилироваться, тесты должны проходить, а
поведение для оператора не должно меняться без отдельного явного решения.

## Цели

- Сделать границы слоёв явными: HTTP, application/usecase, domain helpers,
  storage и integration/legacy adapters.
- Уменьшить повторение в handlers и services без введения абстракций ради
  абстракций.
- Привести логирование к единому `log/slog` и добавить request-level события.
- Перевести важные архитектурные комментарии на русский язык и добавить
  комментарии только там, где они объясняют неочевидное решение.
- Обновить документацию под фактическое состояние проекта.
- Сохранять проверяемость: каждый этап должен иметь понятные команды проверки.

## Не цели

- Не переписывать UI на SPA и не менять выбранный `templ + HTMX` подход.
- Не заменять SQLite, `sqlc`, `goose` или `chi` без отдельной причины.
- Не переносить всё за repository-интерфейсы, если это не снижает связность.
- Не менять бизнес-правила импорта XLSX, протоколов, генерации XML/DOCX,
  auth, CSRF и audit в рамках чистого рефакторинга.
- Не комментировать очевидные CRUD-операции. Комментарии должны объяснять
  причину, инвариант или риск, а не пересказывать строку кода.

## Целевая архитектура

Приложение остаётся modular monolith. Доменные пакеты сохраняют владение своими
workflow:

- `admin`: вход, сессии, CSRF, ограничения попыток входа.
- `audit`: запись и просмотр журнала действий.
- `programs`, `employers`, `people`: справочники и ручные CRUD-процессы.
- `requests`: импорт XLSX, staging rows, применение строк.
- `protocols`: жизненный цикл протоколов и участники.
- `documents`: генерация XML/DOCX и изоляция legacy generator.
- `storage`: SQLite, миграции, транзакции и sqlc-generated queries.

Новые общие пакеты допускаются только если они уменьшают повторение между
несколькими доменами:

- `backend/httpx`: безопасный разбор параметров, единое оформление HTTP-ошибок,
  redirect helpers, обёртка для render errors, request logging middleware.
- `backend/domain`: общий тип field-level validation errors и утилиты для
  человекочитаемых сообщений.
- `backend/platform`: общие инфраструктурные helpers вроде clock/null/log
  только если они не тянут за собой бизнес-смысл.

Пакеты `httpx`, `domain` и `platform` не должны импортировать доменные пакеты.
Зависимости идут от доменов к общим пакетам, а не наоборот.

## Роутинг и wiring

`backend/app/router.go` сейчас совмещает создание зависимостей, middleware,
singleton wiring и регистрацию маршрутов. Целевая структура:

- `backend/app/container.go`: создание общих зависимостей приложения.
- `backend/app/routes.go`: регистрация маршрутов и групп.
- `backend/app/middleware.go`: middleware приложения, включая actor propagation
  и request logging.
- `backend/app/server.go`: server lifecycle и startup config.

Цель не в том, чтобы сделать dependency injection framework, а в том, чтобы
файл с маршрутами читался как карта приложения. Временные singletons для
frozen API (`documents.SetDefaultService`, `admin.SetDefaultHandler`) остаются
изолированными и явно помеченными как compatibility layer.

## Handlers

Handlers должны отвечать за HTTP-детали:

- разбор route/form/query параметров;
- вызов service/usecase;
- выбор redirect/render/error response;
- минимальное security-решение на уровне HTTP, если оно зависит от URL
  ownership, например проверка принадлежности row к request.

Повторяющиеся операции (`ParseForm`, `ParseInt`, `http.Error`,
`templ.Render`) можно вынести в `httpx`, если это не прячет важную ветку
поведения. Ошибки валидации должны возвращаться пользователю как 400 или через
перерисовку формы, storage/system errors должны логироваться и возвращаться как
500 без утечки лишних деталей.

## Services и use cases

Services остаются application layer. Они могут напрямую использовать `sqlc`
queries и `storage.WithTx`. Интерфейсы repository вводятся только там, где
есть реальная альтернатива реализации или существенное упрощение тестов.

Большие workflow разбиваются на приватные use-case helpers:

- `requests`: импорт файла, staging строк, применение строки, audit side
  effects.
- `protocols`: фиксация номера, переходы состояния, участники.
- `documents`: подготовка данных для legacy adapter, generation run lifecycle,
  download/regenerate path.

Каждый helper должен иметь один инвариант, который можно покрыть тестом.

## Логирование

Проект переходит на единый `log/slog`:

- startup, graceful shutdown и ошибки запуска логируются через `slog`;
- seed/dev tooling тоже использует `slog`, если это не усложняет CLI;
- HTTP middleware пишет method, path, status, duration, remote address и actor,
  когда actor уже известен;
- domain services логируют только события, которые невозможно восстановить из
  return error или audit log.

Запрещено логировать значения `.env`, пароли, CSRF/session tokens, содержимое
XLSX, сырые ФИО/СНИЛС/email и исходные имена файлов. Для подозрительных upload
metadata допускаются хэши, как уже сделано в `requests`.

## Комментарии

Новые и обновляемые комментарии пишутся на русском языке. Исключения допустимы
для внешних терминов, имён API и сгенерированного кода.

Комментарии должны покрывать:

- package-level назначение домена;
- архитектурные границы и запреты на импорт;
- security decisions: CSRF, session, actor propagation, rate limit;
- транзакционные инварианты и concurrency;
- legacy adapter contracts;
- причины, почему ошибка логируется или подавляется.

Комментарии не должны объяснять очевидное, например "создаём переменную" или
"возвращаем ошибку".

## Документация

README нужно привести к фактическому состоянию: auth, requests, protocols и
XML/DOCX уже присутствуют, Moodle и полная RBAC-модель остаются вне текущего
реализованного объёма.

Добавляется или обновляется `docs/architecture.md`:

- краткая карта пакетов;
- поток HTTP request -> handler -> service -> sqlc/storage -> audit;
- где живёт legacy generator и почему он изолирован;
- правила логирования и секретов;
- команды проверки после refactor-этапов.

## Этапы реализации

1. Документация и комментарии без изменения поведения.
   Обновить README, добавить `docs/architecture.md`, перевести ключевые
   архитектурные комментарии на русский в местах, которые будут затронуты
   первыми.

2. Application wiring.
   Разнести `backend/app/router.go` на container/routes/middleware, сохранить
   все маршруты и middleware в прежнем порядке. Добавить или подготовить
   request logging middleware.

3. Общие helpers.
   Ввести минимальные `httpx`/`domain` helpers и перенести повторяющиеся
   `parseInt64Param`, `FieldErrors`, render/error helpers. Делать перенос
   пакетами, не массовым поиском-заменой.

4. CRUD-домены.
   Выравнять `programs`, `employers`, `people`: handlers, validation errors,
   audit helpers, русские package comments. Эти домены проще и служат шаблоном
   для более рискованных пакетов.

5. Workflow-домены.
   Аккуратно разложить `requests`, `protocols`, `documents` на более мелкие
   private helpers/use cases. Не менять бизнес-правила и существующие tests.

6. Логирование и production hygiene.
   Унифицировать `slog`, добавить request logging, проверить отсутствие
   секретов/PII в логах, обновить run/check документацию.

7. Финальный аудит.
   Удалить устаревшие внутренние MVP-маркеры, проверить комментарии, пройти
   весь набор тестов и выполнить public-git hygiene scan из `AGENTS.md`, если
   репозиторий готовится к публикации.

## Проверки

Минимальные проверки после каждого этапа:

```bash
make test
make schema-test
```

Если затронуты `*.templ`:

```bash
make templ
make test
```

Если затронуты SQL queries или migrations:

```bash
make sqlc
make schema-test
make test
```

Перед заявлением о public-ready состоянии дополнительно выполняется scan из
`AGENTS.md` по tracked и untracked файлам. Значения секретов и реальные
персональные данные нельзя печатать в чат.

## Риски

- Массовый перенос helpers может ухудшить читаемость, если абстракции станут
  слишком общими. Поэтому helpers вводятся после первого реального повтора.
- Разделение router может случайно поменять порядок middleware. Это проверяется
  существующими router/admin/request tests.
- Workflow-пакеты содержат транзакционные и audit-инварианты. Их нельзя
  рефакторить одновременно с изменением поведения.
- Singleton compatibility layer в `documents` и `admin` нельзя удалить без
  отдельного решения, потому что tests и frozen API на него опираются.

## Критерии готовности

- Роутинг и wiring читаются отдельно от доменной логики.
- Повторяющиеся HTTP и validation helpers вынесены без циклических импортов.
- Все важные комментарии в изменённых местах написаны на русском языке.
- `slog` используется последовательно для runtime logging.
- README и architecture docs соответствуют фактическому состоянию проекта.
- Все обязательные проверки проходят после каждого этапа.
