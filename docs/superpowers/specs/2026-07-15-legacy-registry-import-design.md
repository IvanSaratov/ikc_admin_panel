# Асинхронный импорт промышленного XLSX-реестра

Дата: 2026-07-15
Статус: согласованный дизайн, ожидает ревью письменной спецификации

## 1. Контекст

Заказчик ведёт промышленную базу в многолистовой XLSX-книге. Книга содержит
около 25 тысяч строк на пяти листах (`А`, `Б`, `В`, `П`, `С`) и хранит данные
об организациях, сотрудниках, программах и периодах обучения, протоколах,
результатах, номерах реестра Минтруда и учётных записях учебного портала.

Это первый из двух типов входных файлов. Он используется несколько раз для
переноса исторической базы. Регулярные файлы клиентских заявок являются другим
форматом и позднее получат отдельный профиль импорта `client_request`.

В проекте уже есть пакет `backend/requests`, базовые таблицы `imports`,
`import_rows`, `client_requests` и staging-модель заявки. Текущий parser ожидает
однолистовой файл с колонками `ФИО`, `СНИЛС`, `Email` и `Коды программ`, не
соответствует промышленной книге и не подключён к JSON API. Frontend импортов
сейчас использует mock-сервисы.

## 2. Цели

- Принять промышленную XLSX-книгу через защищённый JSON API.
- Быстро проверить файл до постановки в очередь и вернуть точный HTTP-результат.
- Обработать импорт асинхронно, не удерживая upload-запрос открытым.
- Перенести все доступные бизнес-данные из книги в канонические таблицы.
- Автоматически применить однозначно корректные строки.
- Сохранить спорные строки и причины конфликтов для последующей проверки.
- Не создавать дубли при повторном запросе, перезапуске приложения или загрузке
  обновлённой версии книги.
- Показывать будущему UI очередь, этап, прогресс и результаты через API.
- Не сохранять пароли учебного портала в БД, staging, API, audit или логах.

## 3. Не входит в первый релиз

- UI загрузки, анимация прогресса и таблица проверки.
- Импорт регулярных клиентских заявок.
- WebSocket и Server-Sent Events: UI будет использовать polling.
- Параллельное применение двух импортов.
- Хранение или восстановление паролей учебного портала.
- Внешняя очередь, Redis и отдельный worker-процесс.

## 4. Выбранная архитектура

В Go-приложении запускается один фоновый worker. Состояние очереди, lease,
контрольные точки и прогресс хранятся в SQLite, а не в памяти. Поэтому
перезапуск процесса не теряет задания.

Новый код живёт в отдельном `backend/imports` и не расширяет XLSX parser из
`backend/requests` условными ветками. Границы компонентов:

- HTTP handler — preflight, problem responses и сериализация DTO;
- job service — очередь, lease, статусы, progress и retry;
- legacy parser — workbook в очищенные source rows;
- normalizer/matcher — типизированные значения, issues и canonical candidates;
- applier — идемпотентные транзакционные изменения основных таблиц;
- worker — последовательный вызов фаз и graceful shutdown.

Общие чистые нормализаторы можно вынести из `requests` только после появления
тестов совместимости. Профили `legacy_registry` и `client_request` остаются
разными parser-ами и не делят неявные правила колонок.

Успешная загрузка создаёт ресурс импорта и возвращает `202 Accepted` с
`import_id`, `queue_position` и URL состояния. Разные файлы обрабатываются FIFO.
Одновременно канонические таблицы изменяет только один импорт, что уменьшает
конкуренцию за SQLite writer lock.

Отдельный worker-сервис не используется: для локального приложения в одном MSI
он добавил бы инфраструктуру без практической пользы. Синхронный импорт внутри
HTTP-запроса не используется из-за долгой обработки и невозможности надёжного
восстановления.

## 5. Жизненный цикл задания

Статусы задания:

- `queued` — принято и ожидает worker;
- `processing` — worker владеет действующим lease;
- `completed` — все строки применены либо признаны точными дублями;
- `completed_with_issues` — корректные строки применены, часть строк ожидает
  проверки;
- `failed` — системная ошибка, после которой возможен retry;
- `cancelled` — отменено до начала изменения канонических таблиц.

Текущий этап хранится отдельно:

`preflight → parsing → staging → validating → applying → finalizing`

Переходы:

1. HTTP handler выполняет preflight, сохраняет временный файл и создаёт
   `queued`-задание.
2. Worker атомарно получает самое старое задание и lease.
3. `parsing` проверяет книгу и составляет карту листов и заголовков.
4. `staging` потоково читает строки, удаляет пароль и пакетно сохраняет
   очищенные данные.
5. После полного staging исходный временный XLSX удаляется.
6. `validating` нормализует поля, ищет совпадения и классифицирует строки.
7. `applying` применяет `valid`-строки короткими транзакционными пачками.
8. `finalizing` сверяет счётчики и выставляет итоговый статус.

Worker обновляет heartbeat и контрольную точку после каждой пачки. При старте
приложения задание с истёкшим lease возвращается в работу с последней
подтверждённой точки. Уже применённые строки не выполняются повторно.

Отмена разрешена только до `applying`. После начала применения endpoint отмены
возвращает `409`, поскольку успешные пачки уже являются промышленными данными и
не должны удаляться неявным rollback всей миграции.

## 6. Preflight и HTTP-результаты

Preflight выполняет ограниченную работу, достаточную для отклонения очевидно
неподходящего файла до `202`:

- проверяет auth, роль и CSRF;
- проверяет multipart и наличие ровно одного файла;
- потоково вычисляет SHA-256;
- проверяет лимит upload и доступное временное хранилище;
- проверяет ZIP/XLSX-сигнатуру и безопасно читает central directory;
- ограничивает распакованный размер, число листов, строк, ячеек и длину ячейки;
- открывает workbook metadata и заголовочные строки;
- проверяет наличие поддерживаемых листов и распознаваемого общего ядра колонок;
- сохраняет файл под случайным внутренним именем вне web-root;
- в одной транзакции создаёт задание и метаданные файла.

Начальные защитные лимиты конфигурируемы. Значения по умолчанию:

- upload: 50 MiB;
- суммарный распакованный размер: 256 MiB;
- листов: 32;
- строк: 200 000;
- ячеек: 5 000 000;
- текст одной ячейки: 32 KiB;
- одновременно `queued`/`processing`: 5 заданий.

Ответы upload endpoint:

| Код | Причина |
|---|---|
| `202` | Файл прошёл preflight и задание надёжно создано |
| `400` | Нет файла, несколько файлов или повреждён multipart |
| `401` | Нет аутентифицированной сессии |
| `403` | Недостаточно прав или не прошёл CSRF |
| `413` | Превышен лимит upload |
| `415` | Файл не является XLSX |
| `422` | Книга открывается, но структура не поддерживается |
| `409` | Тот же SHA-256 уже обрабатывается или был успешно импортирован |
| `429` | Достигнут лимит очереди |
| `500` | Неожиданная внутренняя ошибка |
| `503` | БД, очередь или временное хранилище временно недоступны |

`Idempotency-Key` защищает от двойного клика и повторной отправки одного
HTTP-запроса. Повтор с тем же ключом возвращает существующий ресурс: `202`, если
он ещё не завершён, и `200`, если он уже terminal. Одинаковый SHA-256 с другим
ключом возвращает `409` и `existing_import_id`, когда прежний импорт queued,
processing или успешен. Для failed-импорта используется `retry`, пока временный
файл существует; после его expiry тот же SHA-256 можно загрузить заново. Файл с
иным содержимым ставится в очередь, даже если имя совпадает.

Upload, retry и cancel разрешены роли `admin`. Просмотр прогресса и ручное
разрешение строк разрешены `admin` и `operator`. Все mutating endpoints требуют
CSRF и записывают actor из сессии.

## 7. Очередь двух и более файлов

- Разные файлы принимаются до лимита очереди.
- Первое задание обрабатывается, последующие получают FIFO-позицию.
- Ошибка одного задания не останавливает очередь.
- Следующее задание валидируется против состояния БД после успешных пачек
  предыдущего задания.
- Точные совпадения второго файла становятся `duplicate`; новые записи
  создаются; несовместимые совпадения становятся `needs_review`.
- Очередь и позиция вычисляются на сервере и возвращаются в status API.

## 8. Безопасный жизненный цикл файла

Оригинальная книга содержит персональные данные и пароли. Она не сохраняется в
SQLite и не попадает в audit. До завершения staging файл лежит в закрытой
директории под случайным именем с минимальными правами ОС. Исходное имя хранится
только как очищенное имя файла без пути.

Во время staging parser знает колонку пароля, но не копирует её значение ни в
одну структуру, предназначенную для persistence или логирования. После полного
staging временный файл удаляется. При сбое до завершения staging файл хранится
не более 24 часов для retry, затем задание переводится в `failed`, а файл
удаляется. Cancel также удаляет файл.

Обычное удаление файла не гарантирует физическое стирание блоков SSD. Поэтому
рабочая директория приложения должна находиться на защищённом пользовательском
диске, а backup временной директории должен быть отключён.

## 9. Контракт промышленной книги

Профиль требует пять непустых листов `А`, `Б`, `В`, `П`, `С`. Имена
нормализуются по пробелам и регистру. Отсутствующий обязательный лист или
неизвестный непустой лист дают `422`, чтобы импорт не потерял данные молча.
Полностью пустые строки пропускаются; пустая строка не завершает чтение листа.

Общее ядро колонок:

- организация;
- ИНН;
- Ф.И.О.;
- профессия;
- подразделение;
- СНИЛС;
- номер протокола;
- дата протокола;
- оценка;
- период обучения;
- номер в реестре;
- организация/филиал/номер заявки;
- email учебного портала;
- логин;
- пароль, который распознаётся только для удаления.

Лист `В` дополнительно содержит явную программу обучения. Листы `П` и `С`
содержат профильные признаки первой помощи и СИЗ. Header matcher нормализует
точки, дефисы, пробелы, регистр и известные варианты написания. Дополнительные
колонки сохраняются в `extra_fields`, кроме колонок, распознанных как пароль или
другой секрет.

Для чтения используется streaming iterator excelize, а не `GetRows` всей книги.
Числовые ИНН/СНИЛС и даты читаются с учётом отображаемого Excel-формата.
Формулы принимаются только тогда, когда workbook содержит cached value;
исполнение формул на сервере не производится.

## 10. Модель данных

До первого промышленного релиза `001_baseline.sql` является изменяемым
baseline. Отдельная upgrade-миграция и таблица `import_jobs` не создаются:
`imports` непосредственно хранит профиль, этап, lease, heartbeat,
идемпотентность, счётчики, времена выполнения и безопасную ошибку. Профиль имеет
значение `legacy_registry` или `client_request`, а статусы используют единый
job lifecycle.

Изменение baseline намеренно несовместимо со старыми локальными SQLite-файлами.
Перед запуском новой версии developer database или Docker volume удаляются и
создаются заново; промышленной базы на старой схеме не существует.

`import_sheets` хранит:

- `import_id`, имя и порядок листа;
- распознанный профиль листа;
- количество найденных и staged-строк;
- безопасную карту заголовков без значений ячеек;
- статус разбора.

Существующая `import_rows` получает `sheet_name`. Уникальный ключ становится
`(import_id, sheet_name, row_number)`. `raw_data` содержит очищенный JSON без
пароля. Профиль `client_request` явно создаёт строки с sheet name `Заявка`,
чтобы текущий `backend/requests` продолжил работать на общей модели.

`legacy_import_rows` имеет связь один-к-одному с `import_rows` и хранит:

- нормализованные организацию, ИНН, ФИО, СНИЛС, email;
- профессию и подразделение;
- группу и программу обучения;
- начало и окончание периода;
- номер и дату протокола;
- оценку и номер реестра Минтруда;
- историческую ссылку на филиал/заявку;
- логин учебного портала;
- `extra_fields` без секретов;
- статус строки, версию и контрольную точку;
- идентификаторы найденных или созданных канонических сущностей.

`import_row_issues` хранит `field`, стабильный `code`, severity, русское
сообщение и resolution. Полные значения ячеек в issue не дублируются.

`program_aliases` хранит управляемое соответствие профиля листа и
нормализованного текста программе. Alias уникален внутри профиля и всегда
ссылается на заранее созданную программу с заданными кодом и часами.

Статусы строки:

`staged → valid → applying → applied`

Альтернативные результаты: `duplicate`, `needs_review`, `skipped`.

## 11. Каноническое отображение

| Данные XLSX | Каноническая сущность |
|---|---|
| Организация и ИНН | `employers` |
| ФИО, СНИЛС, email | `workers` |
| Работодатель сотрудника | `worker_employers` |
| Профессия и подразделение | `training_records`, а актуальное значение — в assignment |
| Лист, программа, профильные признаки | `program_groups` и `programs` |
| Период обучения | даты протокола и запись обучения |
| Номер и дата протокола | `protocols` |
| Оценка | `protocol_participants.assessment_result` |
| Номер реестра Минтруда | `protocol_participants.mintrud_registry_number` |
| Филиал/номер заявки | `training_records.source_reference` |
| Логин и email портала | `moodle_accounts` |
| Пароль | не сохраняется |

Миграция добавляет недостающие nullable-поля `training_records.department`,
`training_records.source_reference`, `worker_employers.current_department` и
`protocol_participants.assessment_result`. Историческая позиция всегда хранится
на `training_records`. `worker_employers.current_position` и
`current_department` обновляются значениями самой поздней однозначной строки по
дате протокола; одинаковая дата с разными значениями создаёт issue вместо
произвольного выбора. Строка без даты не изменяет актуальные поля assignment,
но сохраняет исторические значения в training record.

Импорт создаёт исторический протокол специальным import-only методом, который
сохраняет исходный номер вместо генерации нового. Протокол получает `fixed`,
если обязательные даты и номер корректны, и `registry_entered`, если все его
активные участники, для которых требуется реестр Минтруда, имеют реестровый
номер. Статус `completed` не выставляется без отдельного подтверждения: наличие
строки в старой книге не доказывает выполнение всех стадий новой админки.

## 12. Правила сопоставления и дедупликации

### Работодатель

- Автоматическое совпадение выполняется только по нормализованному валидному
  ИНН.
- Совпавший ИНН и совместимое название используют существующего работодателя.
- Совпавший ИНН и существенно отличающееся название создают
  `employer_inn_conflict`.
- Отсутствующий или некорректный ИНН требует проверки; одно название не
  является достаточным автоматическим идентификатором.

### Сотрудник

- Автоматическое совпадение выполняется по 11 цифрам СНИЛС.
- Совпавший СНИЛС и совместимое ФИО используют существующего сотрудника.
- Совпавший СНИЛС и другое ФИО создают `worker_snils_conflict`.
- Сотрудник без валидного СНИЛС не создаётся автоматически.

### Программа

- Лист определяет группу, явное название и профильные колонки уточняют
  программу.
- Импорт использует заранее заведённый каталог программ и таблицу стабильных
  aliases.
- Неизвестная или неоднозначная программа создаёт `unknown_program`; worker не
  придумывает код и часы обучения.
- Перед первым промышленным импортом администратор заводит каталог программ и
  aliases. Отсутствие alias не останавливает файл целиком, а оставляет
  соответствующие строки для ручного связывания.

### Протокол и обучение

- Идентичность протокола: группа программы, нормализованный номер и дата.
- Тот же номер в той же группе с другой датой создаёт конфликт.
- Для исторических данных уникальность номера является групповой; глобальный
  unique index номера заменяется на group-scoped constraint.
- Идентичность обучения: сотрудник, работодатель, программа и протокол.
- Существующая идентичная цепочка является дублем, а не новым активным
  обучением. Проверка не использует только текущий `active`-статус.

### Участник и реестр

- Участник связывает обучение с протоколом.
- Отличающиеся оценка или номер реестра у одного участника создают конфликт.
- Пустые необязательные значения не перезаписывают существующие заполненные.

### Учебный портал

- Логин уникален и связывается с сотрудником.
- Тот же логин для другого сотрудника создаёт конфликт.
- Email нормализуется в lowercase.
- Пароль игнорируется без сохранения факта или значения пароля.

## 13. Транзакции и идемпотентность

Staging и applying выполняются пачками. Одна пачка либо полностью committed,
либо полностью откатывается. Успешные предыдущие пачки не откатываются при
поздней системной ошибке и остаются доступными; retry продолжает задание.

Каждая applied-строка хранит ссылки на все созданные или найденные сущности и
отметку checkpoint. Перед созданием worker повторно проверяет source identity и
канонические natural keys. Это защищает от дублей после crash между commit
пачки и обновлением агрегированных счётчиков.

Счётчики задания пересчитываемы из строк и не являются единственным источником
истины.

## 14. API

### Задания

- `POST /api/imports/legacy` — preflight и постановка в очередь;
- `GET /api/imports` — история и очередь с cursor pagination;
- `GET /api/imports/{id}` — status, phase, queue position и счётчики;
- `POST /api/imports/{id}/retry` — retry системной ошибки;
- `POST /api/imports/{id}/cancel` — отмена до `applying`.

### Строки

- `GET /api/imports/{id}/rows` — cursor pagination, фильтры по status, sheet и
  issue code; ответ содержит все данные для таблицы, исправления и resolution;
- `PATCH /api/imports/{id}/rows/{rowId}` — исправить значения или задать связи;
- `POST /api/imports/{id}/rows/{rowId}/apply` — повторно валидировать и применить
  исправленную строку.

Отдельного `GET /api/imports/{id}/rows/{rowId}` в первом релизе нет. Страница из
50 строк загружается одним запросом; backend получает связанные названия и
issues без N+1 запросов. `PATCH` возвращает обновлённую проекцию строки.

Cursor кодирует стабильную сортировку `sheet_order, row_number, id`. Значение
`limit` по умолчанию равно 50, максимум — 200.

Строка имеет `version`. `PATCH` принимает ожидаемую версию; устаревшее изменение
получает `409` и актуальную проекцию строки.

## 15. Ошибки и логирование

API использует единый `application/problem+json` с `code`, русским `detail`,
`trace_id` и безопасными field errors. Ответ не содержит stack trace, полный
raw row или значение секрета.

Ошибки делятся на:

- file-level validation — upload отклоняется либо задание завершается без
  canonical writes;
- row-level validation/conflict — только строка становится `needs_review`;
- transient storage error — ограниченный retry пачки с backoff;
- permanent system error — задание становится `failed` и продолжает очередь;
- panic worker — recover на границе задания, запись безопасной ошибки и
  освобождение lease по timeout.

Логи содержат import ID, phase, sheet, row number, issue code и длительность, но
не ФИО, СНИЛС, ИНН, email, логин, пароль или raw row. Audit фиксирует загрузку,
старт, завершение, retry, cancel, ручное решение и созданные сущности через ID.

Graceful shutdown перестаёт брать новые задания, завершает текущую транзакцию и
оставляет checkpoint для следующего запуска.

## 16. Стратегия тестирования

Реальная промышленная книга не коммитится и не выводится в тестовые логи.
Автотесты генерируют синтетические книги с той же формой и очевидно фиктивными
данными.

### Parser и preflight

- пять профилей листов и варианты заголовков;
- пустые, скрытые, повторяющиеся и полностью пустые строки;
- дополнительные, переставленные, отсутствующие и дублирующиеся колонки;
- numeric/string ИНН и СНИЛС, ведущие нули и formatted dates;
- cached formulas, пустые ячейки и предельная длина;
- повреждённый ZIP/XLSX, zip-bomb и превышение каждого лимита;
- обязательное удаление password column.

### Нормализация и сопоставление

- совпадения и конфликты по ИНН, СНИЛС, программам и протоколам;
- один сотрудник у нескольких работодателей;
- повторные обучения и исторические дубли;
- конфликт оценки и номера реестра;
- Moodle login другого сотрудника;
- сохранение дополнительных полей без секретов.

### Очередь и восстановление

- два разных файла FIFO;
- один файл дважды по SHA-256;
- повтор одного `Idempotency-Key`;
- обновлённая версия после предыдущего импорта;
- лимит очереди;
- failure первого задания не блокирует второе;
- остановка на каждом phase и продолжение по checkpoint;
- истёкший lease;
- cancel до applying и `409` после начала applying;
- ошибка SQLite внутри пачки и полный rollback этой пачки.

### API и безопасность

- все HTTP-коды preflight;
- auth, role и CSRF;
- cursor pagination на 50 строк одним запросом;
- optimistic locking PATCH;
- problem+json без PII;
- отсутствие пароля в таблицах, API, audit и перехваченных логах;
- очистка временных файлов при success, cancel, expiry и error.

### Производительность

Отдельный локальный benchmark создаёт пять листов и около 25 тысяч строк. Он
измеряет время фаз, пик памяти, число SQL-операций и частоту progress updates.
Wall-clock не является хрупким CI assertion; CI проверяет ограниченность памяти
на уровне алгоритма, монотонность прогресса и корректный результат. Локальный
результат фиксируется в отчёте реализации.

## 17. Критерии готовности

- Upload возвращает `202` только после успешного preflight и durable enqueue.
- Два разных файла последовательно обрабатываются без взаимной блокировки
  очереди.
- Корректные строки автоматически создают или связывают все канонические
  сущности.
- Дубли не создают повторных сущностей.
- Спорные строки доступны одним пагинированным запросом для будущей таблицы.
- Ручное исправление можно повторно валидировать и применить через API.
- Crash/restart не теряет задание и не создаёт дубли.
- Пароль отсутствует во всех persistent и observable поверхностях приложения.
- Backend, schema и интеграционные тесты проходят.
- Реальный локальный файл успешно обрабатывается без печати его данных.
