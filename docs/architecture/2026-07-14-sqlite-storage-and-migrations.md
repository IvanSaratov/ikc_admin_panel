# Архитектура SQLite и миграций схемы

**Тип:** постоянное архитектурное решение

**Статус:** migration foundation реализован; performance hardening и MSI остаются отдельными этапами

**Дата:** 2026-07-14

## 1. Контекст

Продукт состоит из одного Go backend и React frontend. Клиентская версия
поставляется в MSI-пакете. Только backend имеет прямой доступ к базе. Целевая
нагрузка — не более 10 одновременно работающих операторов.

Единственная поддерживаемая СУБД — SQLite. Совместимость с PostgreSQL, SQL Server
и несколькими SQL-диалектами не закладывается. Это позволяет осознанно использовать
возможности SQLite и не оплачивать сложность абстракции, которая продукту не нужна.

В проекте уже применяются SQL-миграции Goose, они встраиваются в Go-бинарник и
запускаются до HTTP-сервера. Новая архитектура сохраняет этот подход и доводит его
до production-состояния.

## 2. Цели

- Автоматически создавать полную схему при первой установке.
- Безопасно обновлять существующую базу заказчика.
- Не поставлять SQL-файлы отдельно от совместимого с ними backend.
- Оставить структуру и преобразования данных в версионированном SQL, а не писать
  собственный движок миграций на Go.
- Никогда не запускать HTTP с несовместимой или частично обновлённой схемой.
- Не связывать жизненный цикл пользовательских данных с транзакцией MSI.
- Предоставить технической поддержке команды проверки, миграции и резервного
  копирования.
- Обеспечить согласованную нагрузку в 10 операторов без неконтролируемых блокировок
  SQLite.

## 3. Что не входит в это решение

- Шифрование базы, BitLocker, ACL и управление ключами. Команда решает эти вопросы
  отдельно.
- Поддержка SQL Server, PostgreSQL или переключение СУБД.
- Миграции без остановки и несколько экземпляров backend. У продукта одна служба;
  короткое окно обслуживания при обновлении допустимо.
- Автоматический downgrade схемы при откате или удалении MSI.
- Решение о сохранении или удалении `sqlc`. Слой доступа к данным рассматривается
  отдельно; предложенный механизм миграций не зависит от `sqlc`.

## 4. Компоненты

### 4.1. Встроенные SQL-миграции

SQL-файлы остаются в `backend/migrations/` и компилируются в приложение через
`embed.FS`. В MSI не попадают отдельный Goose CLI и каталог миграций, который можно
потерять, изменить или рассинхронизировать с backend.

Baseline устанавливает SQLite `application_id=0x494B4341` (`IKCA`). До любого
изменения метаданных Goose приложение проверяет эту идентичность и отказывается
запускаться с заполненным файлом без версии либо с pre-release базой без выпущенного
`application_id`.

### 4.2. Единый `Migrator`

Один компонент миграций отвечает за:

- создание Goose `Provider` для SQLite;
- получение текущей и целевой версий;
- перечисление неприменённых миграций;
- применение миграций по порядку;
- структурированный результат с исходной, целевой и фактической версиями;
- ошибки с именем неуспешной миграции.

Используется современный Goose `Provider`, не содержащий глобального состояния.
Legacy-вызовы `SetBaseFS`, `SetDialect` и mutex, защищавший только эту глобальную
конфигурацию Goose, удалены. Отдельный package-level `migratorUpMu` намеренно
сериализует полную операцию `Status` → `Up` внутри одного процесса. Межпроцессное
эксклюзивное владение обеспечивает `OwnerLock`.

Go-код выполняет только оркестрацию. Создание таблиц, индексы, ограничения,
перестроение таблиц и перенос данных описываются в SQL. Go-миграция допускается
только тогда, когда преобразование нельзя разумно и детерминированно выразить в
SQLite SQL, и требует отдельного архитектурного согласования.

### 4.3. `BackupManager`

Компонент резервного копирования создаёт согласованный снимок SQLite через
SQLite-aware механизм, например backup API или `VACUUM INTO`. Простое копирование
только `.db` запрещено: актуальное состояние может находиться в WAL.

Перед миграцией существующей базы создаётся файл с временной меткой и версией
схемы. Приложение хранит три последних pre-migration backup, созданных им самим.
Очистка не затрагивает файлы, созданные администратором.

### 4.4. Блокировка владельца базы

`serve`, `db migrate`, `db verify` и `db backup` получают эксклюзивную процессную
блокировку рядом с базой, например `<database>.lock`. Backend держит её всё время
работы.
ОС освобождает блокировку после завершения или падения процесса; существование
самого lock-файла не считается признаком занятой базы.

Это не позволяет вручную запустить второй backend, migrator, полный integrity check
или backup одновременно с Windows-службой. Только read-only команда `db status`
может выполняться без эксклюзивного владения.

### 4.5. Один бинарник с несколькими командами

Исполняемый файл предоставляет:

- `mintrud-admin serve` — запуск приложения;
- запуск без команды — обратно совместимый alias для `serve`;
- `mintrud-admin db status` — текущая и целевая версии, pending migrations;
- `mintrud-admin db migrate` — применение миграций без запуска HTTP;
- `mintrud-admin db verify` — проверка версии, схемы и целостности;
- `mintrud-admin db backup` — создание согласованной резервной копии.

`serve` и `db migrate` вызывают один `Migrator`. CLI — это диагностический и
аварийный интерфейс, а не второй механизм миграций. Отдельный migrator-бинарник и
новая CLI-библиотека не обязательны.

Все команды используют один алгоритм конфигурации и один абсолютный путь к базе.
Относительный production-путь запрещён, поскольку рабочий каталог Windows-службы
не является надёжной частью конфигурации.

`db status` открывает существующую базу через соединение `mode=ro`. Для
отсутствующего или пустого пути команда сообщает состояние встроенного каталога
миграций и не создаёт файл базы.

## 5. Жизненный цикл установки и обновления

### 5.1. Первая установка

1. MSI устанавливает backend, frontend, конфигурацию и Windows-службу.
2. MSI запускает службу.
3. Backend получает эксклюзивную блокировку и открывает SQLite по абсолютному пути.
4. Для новой базы Goose видит версию 0 и применяет baseline-миграцию.
5. Backend проверяет внешние ключи, целостность и итоговую версию.
6. Выполняется bootstrap приложения.
7. Только после успешных проверок открывается HTTP-порт.

Пустая новая база не требует backup перед созданием схемы.

### 5.2. Обычный перезапуск

1. Backend открывает базу и сравнивает её версию со встроенной целевой версией.
2. Если версии совпадают, SQL не выполняется и backup не создаётся.
3. Backend проверяет совместимость и продолжает запуск.

Таким образом, проверка на каждом старте дёшева: уже применённые миграции повторно
не выполняются.

### 5.3. Обновление через MSI

```text
MSI:     остановить службу -> заменить файлы -> запустить службу
Backend: открыть БД -> backup -> migrate -> verify -> открыть HTTP
```

Подробный порядок:

1. MSI останавливает службу.
2. MSI заменяет файлы и запускает новую версию службы.
3. Backend получает эксклюзивную блокировку базы.
4. До HTTP-старта backend обнаруживает pending migrations.
5. Выполняются `quick_check` и проверка, что схема не новее бинарника.
6. Создаётся согласованный pre-migration backup, после чего backup открывается и
   проходит собственный `quick_check`.
7. Goose применяет неприменённые миграции по порядку.
8. Выполняются `foreign_key_check`, `quick_check` и проверка итоговой версии.
9. В журнал записываются старая и новая версии, длительность и результат.
10. Backend открывает HTTP-порт.

Для продолжительной инициализации Windows-служба остаётся в состоянии
`START_PENDING` и обновляет wait hint/checkpoint, чтобы Service Control Manager не
считал процесс зависшим.

## 6. Обработка ошибок и восстановление

При неуспешной миграции:

- HTTP-порт не открывается;
- процесс пишет номер/имя миграции, исходную и целевую версии и исходную ошибку;
- процесс завершается с ненулевым кодом, а служба остаётся остановленной;
- backup сохраняется для восстановления;
- повторный запуск продолжает с первой неприменённой версии Goose.

Если backup не удалось создать или проверить, миграции не начинаются: backend
пишет ошибку и завершает запуск.

По умолчанию один SQL-файл Goose выполняется в одной транзакции. Аннотация
`NO TRANSACTION` запрещена без отдельного согласования. Если из нескольких файлов
первые успели завершиться, а следующий упал, первые автоматически не откатываются.
Границей восстановления является pre-migration backup.

Приложение не восстанавливает backup автоматически: это может скрыть первичную
ошибку и рассинхронизировать схему с установленным бинарником. Поддержка выбирает
один из двух явных путей:

1. исправление вперёд и повтор миграции;
2. остановка службы, восстановление проверенного backup и установка совместимой
   версии backend.

Старый backend обязан отказаться от запуска, если версия схемы выше максимально
поддерживаемой им. MSI rollback не запускает `Down`. Uninstall удаляет службу и
программные файлы, но по умолчанию сохраняет базу и backup.

## 7. Ответственность MSI

MSI отвечает за:

- установку и замену подписанных программных файлов;
- установку, остановку, запуск и удаление Windows-службы;
- запись несекретной конфигурации и абсолютного пути к базе;
- фиксацию ошибки запуска службы в installer log.

MSI не отвечает за:

- определение текущей версии схемы;
- выполнение SQL;
- выбор миграций;
- downgrade или удаление базы.

В MSI не создаётся Custom Action, изменяющий схему. Windows Installer не может
атомарно откатить одновременно собственные файловые операции и произвольные
изменения SQLite. Для этого потребовались бы отдельные rollback action, передача
контекста/прав, особые условия для upgrade/repair/uninstall и дублирование recovery
логики приложения.

## 8. Правила написания миграций

### 8.1. До первого клиентского релиза

Исходные `001`–`004` были историей разработки, а не безопасной цепочкой
клиентских обновлений. Они объединены в одну чистую baseline-миграцию, которая
сразу создаёт итоговую схему первого релиза.

Это было обязательно, поскольку исходная development-цепочка перестраивала
`client_requests`, добавляла внешний ключ к только что созданной пустой таблице
`imports` и удаляла таблицу, на которую могли ссылаться дочерние записи. Обновление
заполненной базы версии 001 могло завершиться ошибкой, хотя создание пустой базы
проходило успешно.

Baseline замораживается до появления первой клиентской базы.

### 8.2. После первого клиентского релиза

- Применённые миграции неизменяемы: их нельзя редактировать, переименовывать,
  переставлять или повторно использовать их номера.
- Любое изменение схемы или обязательное преобразование данных получает новую
  forward-миграцию.
- Миграция детерминирована и не обращается к сети, внешним сервисам или бизнес-
  сервисам приложения.
- Существующие данные сохраняются либо миграция завершается до разрушительного
  шага.
- Разрушительные изменения выполняются через expand/backfill/verify/contract и
  при необходимости разделяются между релизами.
- Большие backfill выполняются пакетно, а их длительность измеряется до релиза.
- Подстановка переменных окружения в SQL-миграциях выключена.
- Production-восстановление выполняется исправлением вперёд или восстановлением
  backup, а не штатными `Down`-миграциями.

## 9. Настройки и производительность SQLite

Production-база использует:

- WAL;
- `foreign_keys=ON`;
- `busy_timeout=5000`;
- применение connection-scoped pragma к каждому новому соединению через hook
  SQLite-драйвера;
- четыре открытых соединения по умолчанию с возможностью увеличить максимум до
  восьми только по результатам нагрузочного теста;
- короткие write-транзакции без удержания БД во время сетевых вызовов;
- пакетный транзакционный импорт.

SQLite всё равно допускает только одного активного writer. WAL и небольшой pool
нужны, чтобы чтения продолжались параллельно записи, а не для одновременных долгих
write-транзакций.

Правила запросов:

- все растущие списки имеют пагинацию;
- аудит использует keyset pagination вместо глубокого `OFFSET`;
- индексы соответствуют реальным фильтрам и сортировкам;
- точный и prefix-поиск используют обычные индексы;
- произвольный substring/full-text поиск реализуется через FTS5, а не массовые
  `%term%` scan;
- планы критичных запросов проверяются в тестах или release validation.

## 10. Проверка миграций

CI покрывает:

- создание пустой базы;
- повторный запуск на актуальной базе;
- upgrade заполненного fixture каждой ранее выпущенной версии;
- сохранение representative данных и связей;
- foreign-key и integrity checks;
- ошибку внутри транзакционной миграции и последующий retry;
- отказ при схеме новее бинарника;
- создание и восстановление pre-migration backup;
- отказ `db migrate`, `db verify` и `db backup`, пока служба владеет базой.

Upgrade-тест сначала применяет только старый набор миграций, добавляет данные и
только затем запускает новый набор. Единственный запуск всех миграций на пустой
базе считается лишь baseline-тестом.

## 11. Нагрузочная проверка

До первого клиентского релиза выполняется 30-минутный тест:

- 10 параллельных операторов;
- 100 000 работников;
- 20 000 работодателей;
- 1 000 000 строк аудита;
- параллельные list/detail/search/update/audit операции;
- один репрезентативный массовый импорт.

Критерии выпуска:

- нет необработанных `SQLITE_BUSY`, integrity или foreign-key ошибок;
- нет потерянной или частично зафиксированной бизнес-операции;
- p95 обычных CRUD/list/search запросов не превышает 500 мс; загрузка файлов,
  завершение импорта и генерация документов измеряются отдельно;
- WAL не растёт монотонно при наличии возможностей для checkpoint;
- число соединений и память остаются ограниченными весь тест.

## 12. Наблюдаемость и поддержка

Журнал миграции содержит:

- абсолютный путь к базе без содержимого данных;
- текущую и целевую версии;
- список pending migrations;
- путь и результат backup;
- длительность каждой миграции и всего процесса;
- результат integrity check;
- итоговый статус.

CLI возвращает ненулевой код при ошибке конфигурации, блокировки, backup,
несовместимой версии, миграции или проверки целостности. Интерактивная команда
пишет читаемый результат в консоль, Windows-служба использует структурированные
application logs.

## 13. Отклонённые варианты

### MSI Custom Action как владелец миграции

Отклонён из-за несовместимых границ транзакций MSI и SQLite, сложного rollback и
особых условий для upgrade/repair/uninstall.

### Ручная миграция как штатный путь

Отклонена: шаг можно пропустить, выполнить для неправильного пути или запустить
несовместимые версии приложения и схемы. CLI остаётся аварийным инструментом.

### SQL-файлы рядом с бинарником

Отклонены: их можно потерять или рассинхронизировать с backend. Встроенные миграции
делают бинарник и целевую схему одной единицей релиза.

### ORM `AutoMigrate`

Отклонён: он скрывает точную последовательность преобразования существующих
клиентских данных и ухудшает воспроизводимость/review.

### Отдельный migrator-бинарник

Отклонён: для одной локальной службы он усложняет упаковку и согласование версий,
не давая преимуществ перед командами основного бинарника.

## 14. Критерии готовности реализации

- Первый релиз содержит одну проверенную baseline-миграцию.
- Backend создаёт или обновляет SQLite до открытия HTTP.
- Схема новее бинарника блокирует запуск.
- `db status`, `db migrate`, `db verify` и `db backup` используют тот же core, что
  и startup.
- Для существующей базы с pending migrations создаётся согласованный backup.
- Одновременно базой владеет только один `serve`/mutating CLI процесс.
- Ошибка миграции оставляет службу недоступной, понятный лог и recovery backup.
- В MSI нет Custom Action, изменяющего базу.
- Проходят populated-upgrade, backup/restore, failure/retry, idempotence и load
  tests.

## 15. Основания решения

- [Goose Provider](https://pressly.github.io/goose/documentation/provider/) —
  изолированный provider, `embed.FS`, управление версиями и запуск миграций.
- [Goose SQL annotations](https://pressly.github.io/goose/documentation/annotations/)
  — транзакции SQL-миграций и явный режим `NO TRANSACTION`.
- [Microsoft: Rollback Custom Actions](https://learn.microsoft.com/en-us/windows/win32/msi/rollback-custom-actions)
  — ограничения отката внешних изменений, выполненных MSI Custom Action.
- [SQLite: Write-Ahead Logging](https://www.sqlite.org/wal.html) — параллелизм
  readers/writer и роль WAL.
- [SQLite Backup API](https://www.sqlite.org/backup.html) — создание согласованной
  копии работающей базы.
